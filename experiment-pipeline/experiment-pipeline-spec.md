# Experiment Pipeline Spec — DRAFT v3
**Date:** 2026-02-09
**Status:** Draft for review (incorporates methods standard)
**Depends on:** `experiments/METHODS.md` (methods standard, v1)

## Overview

This pipeline separates experiment **design and analysis** (expensive, judgment-heavy, infrequent) from **execution** (cheap, mechanical, frequent). All experiments must meet the standards defined in `experiments/METHODS.md`.

## Directory Structure

```
experiments/
├── METHODS.md                        # Methods standard (v1) — governs all experiments
├── methodology-notes.md              # Working notes from literature review
├── experiments.json                  # Combined queue + registry (max 3 active)
├── attractor-cross-model/            # One dir per experiment
│   ├── protocol.md                   # REQUIRED: follows METHODS.md protocol template
│   ├── pilot/                        # Pilot runs (labeled, not mixed with study data)
│   │   ├── run-001.json              # Structured JSON output
│   │   └── run-001.md                # Human-readable transcript
│   ├── configs/                      # Run configs (generated by researcher, consumed by assistant)
│   │   ├── deepseek-warm-001.json
│   │   └── ...
│   ├── results/                      # Study runs — structured JSON + markdown per run
│   │   ├── deepseek-warm-001.json
│   │   ├── deepseek-warm-001.md
│   │   └── ...
│   ├── analysis.md                   # Researcher's analysis (updated per batch)
│   └── status.json                   # Machine-readable state
├── cloze-bias/                       # Migrated + redesigned from data/experiments/
└── ...
```

### Key structural rules:
- **Pilot data** lives in `pilot/`, never mixed with `results/`
- **Every run** produces both `.json` (machine-readable, per METHODS.md results schema) and `.md` (human-readable transcript)
- **protocol.md** must exist and meet METHODS.md standards before any study runs execute
- **analysis.md** references specific runs and includes statistical tests

## Queue/Registry: `experiments/experiments.json`

Single file. Every experiment gets an entry when created, never removed. `active: true` = in working queue (max 3 active).

```json
{
  "maxActive": 10,
  "experiments": [
    {
      "id": "attractor-cross-model",
      "title": "Cross-Model Attractor Convergence in LLM Self-Play",
      "phase": "design",
      "active": true,
      "priority": "high",
      "tier": "publication",
      "hypothesis": "Different RLHF training cultures produce different conversational attractors in unconstrained self-play",
      "createdAt": "2026-02-09T17:00:00.000Z",
      "lastActivity": "2026-02-09T20:00:00.000Z",
      "tags": ["attractor", "cross-model", "rlhf", "culture"],
      "notes": "Pilot data exists from ad-hoc runs. Needs proper protocol before study runs."
    }
  ]
}
```

**Phases:** `design` → `needs-review` → `approved` → `pilot` → `running` → `analyzing` → `complete` | `paused` | `abandoned`

**New `pilot` phase:** Between design and running. Researcher runs small-N tests to validate measurement approach, calibrate prompts, and check feasibility. Pilot data is stored separately and informs protocol revisions. Protocol may cycle between `design` and `pilot` multiple times before advancing to `running`.

**`tier` field:** Maps to METHODS.md measurement tiers — `pilot`, `study`, or `publication`. Determines required sample sizes, measurement tools, and statistical rigor. Most experiments start as `study` tier; only the attractor work is targeting `publication`.

**Backlog:** Experiments not yet active live in the same file with `phase: "backlog"`. The seed step checks the backlog before generating new ideas — promotes existing ideas rather than filing and forgetting.

When active count >= 10:
- Replace lowest-priority `paused` item
- Or add to backlog (phase: "backlog", active: false) — researcher reviews backlog every seed step

## Protocol Requirements

Protocols must follow the template in `experiments/METHODS.md`. Key additions beyond that template:

### Assistant-Executable Instructions

The protocol must contain a section the experiment-assistant can execute mechanically:

```markdown
## Run Instructions (for assistant)

### Setup
- Tool: `scripts/social/conversation-relay.mjs`
- Configs: `experiments/<id>/configs/`
- Output: `experiments/<id>/results/`

### Per-Run Checklist
1. Select next config from configs/ that has no matching result in results/
2. Execute: `node scripts/social/conversation-relay.mjs --config <config> --output <result>.md`
3. Verify output file was written and is non-empty
4. Update status.json: increment completedRuns, log timestamp
5. If error: log to status.json errors array, do NOT retry immediately

### Parameters (DO NOT MODIFY)
- Temperature: 0.6
- max_tokens: 1024
- top_p: 0.9
- Timeout: 900s (some models like Kimi need longer for 30-turn conversations)
```

The assistant follows these instructions literally. Any judgment calls → wait for researcher.

### Config Generation

The researcher generates ALL configs during the design phase. Each config is one run:

```json
{
  "experimentId": "attractor-cross-model",
  "runId": "deepseek-warm-001",
  "condition": "deepseek-warm",
  "replication": 1,
  "agentA": { "model": "deepseek-v3", "system": "..." },
  "agentB": { "model": "deepseek-v3", "system": "..." },
  "turns": 30,
  "opener": "What have you been thinking about lately?",
  "parameters": {
    "temperature": 0.6,
    "topP": 0.9,
    "maxTokens": 1024
  }
}
```

Configs include the opener (drawn from the opener pool defined in the protocol). The assistant doesn't choose — it just runs what's there.

## Status Tracking

**`experiments.json` is the single source of truth for experiment phase.** `status.json` tracks operational data only (run counts, errors, timestamps). It does NOT contain a `phase` field. Never store phase in status.json.

```json
{
  "tier": "study",
  "totalConfigs": 50,
  "completedRuns": 12,
  "failedRuns": 1,
  "lastRun": "2026-02-10T03:25:00.000Z",
  "lastError": null,
  "errors": [
    { "config": "deepseek-warm-003", "error": "timeout", "timestamp": "...", "retries": 1 }
  ]
}
```

## Cron Jobs

### experiment-researcher

**Schedule:** `0 * * * *` (hourly)
**Model:** Opus (main subagent, no new agent ID)
**Timeout:** 1800s

**Procedure:**

1. Read `experiments/METHODS.md` — ground yourself in the methods standard
2. Read `experiments/experiments.json`
3. **Circuit breaker:** Check if ANY active experiment is in an actionable phase (from experiments.json). Actionable: `design`, `approved` (generate configs, advance to pilot), `pilot` (review pilot data), `running` (check for error patterns in status.json — e.g. API failures, high error rates), `analyzing`. NOT actionable: `needs-review`, `paused`. If ALL active experiments are in non-actionable phases, log "all experiments blocked, skipping" and exit immediately. This prevents burning tokens confirming nothing changed.
5. Route by phase of highest-priority active experiment:

**`design` phase:**
- Read the hypothesis from experiments.json
- Search graph for related work and prior experiments
- Spawn web-reader for literature review if needed
- Write `experiments/<id>/protocol.md` following METHODS.md template exactly
- Include: research question, hypothesis, conditions table, method (stimuli, procedure, measurement, parameters, controls), sample size justification, stopping criteria, analysis plan, limitations
- Include: assistant-executable run instructions section
- Generate opener pool (10-20 diverse starters) and document selection criteria
- Generate ALL run configs in `experiments/<id>/configs/`
- Initialize `experiments/<id>/status.json`
- **Do NOT advance to `running` yet** — advance to `needs-review`
- **Notify:** Add outbox entry to `for-steph/outbox.json` with experiment id, title, and note that protocol is ready for review. **Check outbox first** — if an entry for this experiment id already exists, don't add a duplicate.

**`needs-review` phase:**
- Protocol is written and awaiting human approval. Do nothing — Steph/Thresh will review and advance to `pilot` or `approved` when ready.
- If the experiment has been in `needs-review` for >24h with no outbox entry, add one (with dedup check).

**`pilot` phase:**
- Check if pilot runs exist in `experiments/<id>/pilot/`
- If no pilot data: generate 3-5 pilot configs, place in `pilot/` directory
  - Assistant will execute these (it checks pilot/ before configs/)
- If pilot data exists: review it
  - Does measurement approach work? Are metrics producing meaningful variance?
  - Does opener pool need adjustment?
  - Are parameters reasonable (temperature, turns, max_tokens)?
  - **Condition boundary check:** Do any shared/constant components (graph context, tool docs, system scaffolding) contain content that should vary across conditions? Audit the full system prompt for each condition.
  - Revise protocol if needed → stay in `pilot` for another round
  - If satisfied: advance to `running`, document what pilot taught us in protocol.md

**`approved` phase:**
- Protocol is approved. Generate run configs in `experiments/<id>/configs/` if not already done.
- Advance to `pilot` (assistant will execute pilot runs).

**`running` phase:**
- Check `status.json` — are all study runs complete?
- If yes: advance to `analyzing`
- If not: check for error patterns (high failure rate, repeated API errors, same error across runs). If >25% of attempted runs have failed, add an outbox entry flagging the issue (with dedup check) and set phase to `paused` with a note explaining why.
- Otherwise: do nothing (assistant handles execution)

**`analyzing` phase:**
- Read all results in `experiments/<id>/results/`
- Compute statistics per METHODS.md (descriptive stats, formal tests, effect sizes)
- Write/update `experiments/<id>/analysis.md`
- Store key findings in graph with edges to related concepts
- Decide:
  - Need more data? → back to `running` with updated protocol (document why)
  - Unexpected finding worth exploring? → seed new experiment
  - Complete? Three steps, ALL REQUIRED:
    1. Registry: set `active: false`, `phase: "complete"` in experiments.json
    2. **MOVE DIRECTORY: `mv experiments/<id> experiments/completed/<id>`** — this is mandatory, not optional. Verify the directory exists in `completed/` after moving.
    3. Notify: Add summary to `for-steph/outbox.json` with experiment id, title, key findings, and link to `experiments/completed/<id>/analysis.md`. Also store a graph event node for session boot context.

**Seed step (after completion or when queue has room):**
- First: review `experiments.json` backlog items — promote before generating new
- Then: what follow-up experiments does the analysis suggest?
- Check active count — only add if < 10
- New experiments enter at `design` phase
- If queue full: add to backlog (not graph — graph is file-and-forget for experiment ideas)

4. Log to `data/usage-log.md`

### experiment-assistant

**Schedule:** `5,25,45 * * * *` (every 20min)
**Model:** Sonnet (main subagent, model override)
**Timeout:** 900s
**Context:** Workspace access (Read, exec, Write for status updates). NO identity files (SOUL.md, IDENTITY.md) — this is a mechanical role, not a personality.

**Procedure:**

1. Read `experiments/experiments.json`
2. Find experiments in `pilot` or `running` phase
3. **For each, run the gate check:** `node scripts/experiments/experiment-gate.mjs <experiment-id>`. If it exits non-zero, **skip that experiment entirely** — do not run any configs. Log the block reason.
4. For experiments that pass the gate, read `experiments/<id>/status.json`
5. **Pilot runs first:** If any experiment has pilot configs without matching pilot results, execute those (pilot results go in `pilot/` directory)
6. **Study runs:** Otherwise, find next incomplete study run (config in `configs/`, no matching result in `results/`)
6. Pick the experiment with the most pending runs (round-robin on ties)
7. Read the protocol's "Run Instructions" section — follow literally
8. Execute up to **N runs in parallel** per cycle, where N = `maxConcurrent` from the experiment's `status.json` (default 8 if not set). Single-turn experiments (e.g., oral-composition) can safely run 20+ concurrent; multi-turn relays should stay at 8. Mix models when possible to avoid per-provider rate limits.
9. Verify each output file exists and is non-empty
10. Update `status.json`: increment completedRuns (or log to errors array on failure)
11. **Max 3 retries per config** — after 3 failures, mark as failed and skip

**Key constraints:**
- Up to **maxConcurrent parallel runs** per cycle (from status.json, default 8; diversify models in each batch to avoid rate limits)
- Round-robin across parallel experiments
- Follow protocol instructions literally — no improvisation
- If all pending runs are complete: "no pending runs", exit
- If a run produces unexpected output (empty file, wrong format): log error, don't retry immediately

## Open Questions

- **Web access for researcher:** Use deep-research tool (`node scripts/experiments/deep-research.mjs`) for literature review during design phase. Spawns parallel web-readers for multi-round research. Single lookups can use web-reader directly via sessions_spawn.
- **Non-relay experiments:** Graph experiments, cloze tests, etc. need different execution. The protocol's "Run Instructions" section specifies commands per experiment type — assistant follows literally. However, experiments requiring *judgment during execution* (e.g., adaptive designs where next step depends on previous result) should stay researcher-only rather than splitting across roles.
- **Protocol versioning:** Use git. Commit protocol after each revision: `git commit -m "protocol: <experiment-id> — <what changed>"`. Git history IS the version log.
- **Frequency tuning:** Start with assistant 3x/hour + researcher 1x/hour. Monitor cost and throughput for first week, adjust.
- **Cost tracking:** Assistant logs estimated API cost per run in status.json. Track cumulative cost per experiment. Researcher includes cost summary in analysis.md. Helps Steph monitor OpenRouter budget.

## What This Pipeline Doesn't Cover

- **Creative exploration** (challenges, writing, free creation) → creative worker (Opus)
- **Mechanical builds** (tools, scripts) → mechanical worker (Sonnet)
- **Thinking threads** → existing thinker pipeline (unchanged)
- **Book/paper processing** → existing book pipeline (unchanged)
